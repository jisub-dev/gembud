## 📋 Executive Summary (요약)

Gembud는 게임 파티원 모집 플랫폼으로, **온도 기반 평판 시스템**과 **AI 매칭 추천**을 핵심 차별화 요소로 하는 소셜 게이밍 서비스입니다.

**핵심 지표**:
- **백엔드**: 80개 Java 파일, 136개 테스트 케이스
- **API**: 70+ REST 엔드포인트 + WebSocket
- **데이터베이스**: 12개 Flyway 마이그레이션
- **개발 완성도**: 백엔드 MVP 완료 (Phase 1-10), 프론트엔드 Phase 1 완료

**시장 포지셔닝**: 당근마켓의 온도 시스템 + Discord의 커뮤니티 기능을 게이밍 영역에 적용

## 🎯 1. 서비스 개요

### 1.1 핵심 가치 제안 (Value Proposition)

```
"게임 실력보다 매너가 중요한 파티원을 찾고 싶을 때"
```

**타겟 유저**:
- 게임을 즐기지만 독성 플레이어와 매칭되는 것을 싫어하는 20-30대
- 비슷한 실력과 온도를 가진 파티원을 찾는 유저
- 롤, 배그, 발로란트 등 팀 기반 게임 플레이어

**핵심 차별점**:
1. **온도 시스템** (0-100°C): 게임 후 평가 기반 평판 점수
2. **AI 매칭 추천**: 필터 매칭(40점) + 온도 호환성(30점) + 과거 평가(20점) + 방장 보너스(10점)
3. **하이브리드 매칭**: 공통 옵션(티어, 포지션) + 게임별 세부 옵션
4. **실시간 소통**: WebSocket 채팅 + 알림

## 🏆 2. 경쟁사 비교 분석

### 2.1 주요 경쟁사

| 서비스 | 특징 | 강점 | 약점 |
|--------|------|------|------|
| **Discord** | 범용 커뮤니케이션 플랫폼 | 강력한 커뮤니티 기능, 보이스 채팅 | 게임별 매칭 기능 없음, 독성 유저 필터링 부족 |
| **게임 내장 매칭** (롤, 배그 등) | 게임사 제공 | 원클릭 매칭, 유저 풀 큼 | 랜덤 매칭, 매너 필터링 없음 |
| **오픈채팅 (카카오톡)** | 한국 특화 모바일 | 접근성 높음 | 구조화된 매칭 없음, 신뢰도 검증 부족 |
| **플레이파티** | 국내 게임 파티 모집 | 게임별 방 생성 | UI/UX 구식, 평판 시스템 약함 |

### 2.2 Gembud의 경쟁 우위

#### ✅ 강점

1. **온도 시스템 (평판 경제)**
   - 당근마켓 스타일의 검증된 UX
   - 30°C 미만 시 방 생성 제한 → 독성 유저 자연 배제
   - 3차원 평가 (매너, 실력, 소통) → 다각도 평가

2. **AI 기반 추천 알고리즘**
   - 100점 만점 매칭 점수 + 한글 추천 사유
   - 과거 평가 기록 활용 → 신뢰도 향상
   - 온도 호환성 30점 비중 → 비슷한 성향 매칭

3. **하이브리드 매칭 옵션**
   - 공통 옵션 (티어, 포지션) + 게임별 옵션
   - JSON 저장 방식 → 새 게임 추가 용이
   - 유연한 필터링

4. **기술적 우수성**
   - Spring Boot 3 + React 18 최신 스택
   - 136개 테스트 케이스 → 높은 안정성
   - Redis 캐싱 → 성능 최적화
   - WebSocket STOMP → 실시간 통신

5. **확장 가능한 아키텍처**
   - Flyway 마이그레이션 → DB 버전 관리
   - ADR 문서화 → 의사결정 추적
   - 계층 구조 명확 → 향후 마이크로서비스 전환 용이

#### ⚠️ 약점

1. **네트워크 효과 부족**
   - 신규 서비스로 초기 유저 풀 작음
   - Discord 대비 커뮤니티 기능 제한적
   - 보이스 채팅 미지원 (Phase 11+ 필요)

2. **게임사 API 미연동**
   - 실제 게임 티어 검증 불가 (Phase 12 예정)
   - 유저 자가 입력에 의존 → 허위 정보 가능성

3. **모바일 앱 부재**
   - 웹 전용 → 모바일 편의성 낮음
   - 게이머들의 모바일 사용 선호도 고려 필요

4. **수익 모델 부재**
   - 광고, 구독, 프리미엄 기능 등 수익화 전략 미정의
   - 지속 가능성 불확실

5. **콘텐츠 부족**
   - 5개 게임만 지원 (롤, 배그, 발로란트, 오버워치2, FIFA온라인4)
   - 스팀 게임 확장 필요 (Phase 11 예정)

## 🔍 3. 핵심 로직 점검

### 3.1 온도 시스템 (Temperature System)

**로직 흐름**:
```
게임 종료 → 평가 작성 (매너/실력/소통 각 5점)
→ 평균 점수 계산
→ 온도 증감 (-0.5°C ~ +0.5°C)
→ 30°C 미만 시 방 생성 제한
```

**점검 결과**:
- ✅ **강점**: 간단하고 직관적, 당근마켓 유사 UX
- ✅ **강점**: 3차원 평가로 다각도 피드백
- ⚠️ **리스크**: 평가 조작 가능성 (친구끼리 높은 점수)
- ⚠️ **리스크**: 온도 회복 어려움 (한번 떨어지면 복구 힘듦)

**개선 제안**:
1. **평가 가중치 도입**: 온도 높은 유저의 평가에 더 높은 가중치
2. **이상치 탐지**: 비정상적으로 높은 평가 패턴 감지 및 필터링
3. **온도 회복 이벤트**: 연속 좋은 평가 시 보너스 온도 부여

### 3.2 AI 매칭 추천 알고리즘

**점수 계산식** (총 100점):
```java
- 필터 매칭: 40점 (티어, 포지션, 게임 옵션 일치도)
- 온도 호환성: 30점 (나와 방장의 온도 차이)
- 과거 평가: 20점 (이전에 함께한 유저의 평가)
- 방장 온도 보너스: 10점 (방장 온도가 높을수록)
```

**점검 결과**:
- ✅ **강점**: 다차원 점수로 종합 판단
- ✅ **강점**: 한글 추천 사유 생성 (UX 친화적)
- ✅ **강점**: 온도 호환성 30점 → 독성 유저 회피
- ⚠️ **리스크**: 신규 유저는 과거 평가 없어 점수 낮음
- ⚠️ **리스크**: 필터 가중치 고정 → 유저별 선호도 미반영

**개선 제안**:
1. **협업 필터링 도입**: "이 방을 좋아한 유저들은 이 방도 좋아했습니다"
2. **머신러닝 기반 가중치 최적화**: 실제 매칭 성공률 데이터로 학습
3. **신규 유저 부스트**: 최초 10게임은 평가 점수에 가중치

### 3.3 신고 시스템

**로직 흐름**:
```
신고 생성 → 중복 체크 (동일 방 내)
→ 관리자 검토 (PENDING → REVIEWED)
→ 처리 완료 (RESOLVED)
→ 알림 전송
```

**점검 결과**:
- ✅ **강점**: 중복 신고 방지
- ✅ **강점**: 상태 관리 명확 (PENDING/REVIEWED/RESOLVED)
- ⚠️ **리스크**: 자동 처리 없음 → 관리자 부담 증가
- ⚠️ **리스크**: 허위 신고 방지 메커니즘 부족

**개선 제안**:
1. **자동 경고 시스템**: 3회 이상 신고 시 자동 7일 정지
2. **허위 신고 패널티**: 허위 신고 확인 시 신고자에게 온도 차감
3. **신고 사유 분류**: 욕설, 게임 방해, 허위 정보 등 카테고리화

### 3.4 실시간 채팅 (WebSocket)

**메시지 저장 전략**:
```
- ROOM_CHAT: 저장 안 함 (실시간만)
- GROUP_CHAT: 최근 100개만 저장
- DIRECT_CHAT: 전부 저장
```

**점검 결과**:
- ✅ **강점**: 차등 저장으로 DB 부담 감소
- ✅ **강점**: STOMP 프로토콜로 표준화
- ⚠️ **리스크**: ROOM_CHAT 저장 안 함 → 분쟁 시 증거 없음
- ⚠️ **리스크**: 동시 접속자 증가 시 메모리 부담

**개선 제안**:
1. **ROOM_CHAT 최소 저장**: 신고 대비 최근 50개만 저장
2. **메시지 압축**: 오래된 채팅은 S3로 이동
3. **Redis Pub/Sub 활용**: 메시지 브로커로 확장성 향상

## 📊 4. 기술 스택 평가

### 4.1 백엔드 (Spring Boot 3.2.x)

**강점**:
- ✅ Java 17 최신 LTS 버전
- ✅ Spring Security + JWT + OAuth2 완전한 인증 체계
- ✅ Flyway로 DB 버전 관리
- ✅ Redis 캐싱으로 성능 최적화
- ✅ 136개 테스트 케이스 (높은 안정성)

**약점**:
- ⚠️ 모놀리식 아키텍처 → 트래픽 증가 시 수평 확장 어려움
- ⚠️ JVM 기반 → 메모리 사용량 높음

**점수**: 9/10 (MVP 단계에서는 최적)

### 4.2 프론트엔드 (React 18 + TypeScript)

**강점**:
- ✅ Vite 5.x로 빠른 빌드
- ✅ TanStack Query v5로 서버 상태 관리
- ✅ Zustand로 간결한 클라이언트 상태 관리
- ✅ Tailwind CSS + 게이밍 다크모드 디자인
- ✅ TypeScript strict 모드 (타입 안전성)

**약점**:
- ⚠️ 프론트엔드 개발 Phase 1만 완료 (진행률 10%)
- ⚠️ 모바일 앱 부재 (React Native 전환 필요)

**점수**: 7/10 (완성도 낮음, 하지만 방향성 우수)

### 4.3 데이터베이스 (PostgreSQL 15)

**강점**:
- ✅ ACID 보장 (평가, 온도 계산 정확성)
- ✅ JSON 타입으로 게임별 옵션 유연 저장
- ✅ 복잡한 쿼리 지원 (매칭 필터링)

**약점**:
- ⚠️ 읽기 부하 시 병목 가능 → Read Replica 필요

**점수**: 9/10

## 📈 7. 시장 분석

### 7.1 TAM/SAM/SOM

- **TAM** (Total Addressable Market): 전 세계 게이머 30억 명 × $10/년 = $300억
- **SAM** (Serviceable Addressable Market): 한국 PC 게이머 1,000만 명 × $50/년 = $5억
- **SOM** (Serviceable Obtainable Market): 초기 5년 내 1% 점유 → $500만 (50억원)

### 7.2 경쟁 구도

```
독점적 우위 영역: "온도 기반 평판 + 게임 매칭"
└─ 당근마켓 (평판) + Discord (게이밍) 융합 포지셔닝
```

### 7.3 성공 시나리오

**Year 1**: 10만 명 가입, 일 활성 유저 5,000명
**Year 2**: 50만 명 가입, 일 활성 유저 2만 명 (수익: 광고 + 구독)
**Year 3**: 200만 명 가입, 일 활성 유저 10만 명 (게임사 제휴)

## 📝 9. 결론

Gembud는 **"온도 기반 평판 시스템"**과 **"AI 매칭 추천"**을 핵심으로 하는 차별화된 게임 파티 모집 플랫폼입니다.

**기술적으로는 우수한 수준**이며, Spring Boot 3 + React 18 최신 스택과 136개 테스트 케이스로 안정성을 확보했습니다. 특히 **온도 시스템은 게임 커뮤니티의 독성 문제를 해결할 수 있는 독창적인 솔루션**입니다.

**현재 가장 큰 과제는**:
1. 초기 유저 확보 (네트워크 효과)
2. 수익 모델 정의
3. 모바일 앱 개발

이 3가지를 **12주 이내에 해결**한다면, **투자 유치 및 시장 진입 가능성이 높습니다**.

**최종 권고**: **조건부 GO** (수익 모델 정의 후 본격 론칭)
